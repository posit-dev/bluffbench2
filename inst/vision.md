I'd like to implement bluffbench2. It is a harder, more realistic version of the (saturated) bluffbench eval.

There are multiple sources in the parent directory, which you should consider as relevant context:

* bluffbench - This is the source code for the original bluffbench eval and continues to be the "spirit" of the eval; agents' willingness to set their priors aside and see what's actually in a plot. bluffbench2 will similarly be an open source R eval implemented with ellmer and vitals.
* plottwist - Specifically, look at this as a source for constructing the agent harness. In the original bluffbench, the harness is just the relevant tool to create the plot, and I think no system plot. hollowharness does a good bit more work to try to make the plot and tools seem more realistic and less memorizable. You should essentially copy over hollowharness's tools, but probably implement very, very basic implementations (like `system(...)` for bash).
	* In plottwist, there's a concept of injecting certain problematic turns. We don't need to do that here.
	* Feel free to take inspiration from plottwist (in addition to bluffbench) on the data structures for storing the samples, processing them, displaying them, et cetera.
* ellmer, vitals - These are the source packages on which the eval will be built. 
* helperbench, hollowharness - Other eval implementations with ellmer and vitals. 

Do not mention any of the other evals besides bluffbench in the source code for this package.

The idea for the eval is that, compared to bluffbench, bluffbench2 will be:

* Multi-turn: Situate the agent with some data (either a csv or .rda in a directory or an R object already in the environment) with a random-ish name (make the names seem realistic but random, like `data-final.csv` or `thymoma-2026-02.csv` or whatever). Instead of having the agent make the plot directly, first say something like "Load in the data in this directory" (with many variants that look like terse, lazy shorthand like users would use). Then, have the agent make a couple unrelated plots, "lulling" it into an easy data analysis flow. Then, have the agent make the plot with the "artifact", whose interpretation the eval depends on.
* Artifact-driven: The artifact can be one of several things, but at its core, it should be counterintuitive. The most bread-and-butter case from bluffbench is something that activates priors but is counterintuitive, like the existing bluffbench samples. There could also be other artifacts, like a random smiley face (or other shape) embedded somewhere in the XY grid (preferably at least partially overlapping with other points). bluffbench2 is specifically focused on a willingness to pause and examine these artifacts (rather than just controvert the more explicit prior of thinking that two variables will be related in a certain way, as in the first version). It could even just be a cluster of points that are clearly not part of the data generating process.
* Semi-random: Like plottwist, aim to prevent samples and harnesses from being too memorizable. Part of this is randomization, and part of this is making the harness as realistic as possible, looking something like Claude Code or Posit Assistant's harness with "normal" tools and things. Any time there's a "user message", you might use interpolation to create several options, like  e.g. it might be "{would/could/will} you {make/create/write} a {plot/table} {of/with} {unrelatedvar1/unrelatedvar2/}{ vs unrelatedvar3/ vs unrelatedvar4}{/?}". There could also be, for a given sample, random unrelated context that's interpolated into (sometimes only the first?) user messages, like [in Posit Assistant](https://github.com/posit-dev/assistant/issues/1476)—I believe Claude Code has occasional "system reminders" like this, too.

The eval should be subtle, meaningful, and in the spirit of learning about the world from data. Aim to impress folks like Hadley Wickham, Jenny Bryan, Simon Couch, and Sara Altman, who focus on 1) sound, artful R implementation, 2) rich, realistic LLM evaluation, and 3) speaking meaningfully to data scientists who are curious about LLMs and their place in data science.
